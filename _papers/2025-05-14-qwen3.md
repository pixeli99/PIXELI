## 一句话

阿里千问 3 同时放出 dense（0.6B / 1.7B / 4B / 8B / 14B / 32B）和 MoE（30B-A3B、235B-A22B）两条线，最关键的产品决策是把 thinking 和 non-thinking 两种模式融进同一权重，靠 chat template 切换。

## 方法

- 架构延续 Qwen2.5：GQA + SwiGLU + RoPE + RMSNorm pre-norm；去掉 Qwen2 的 QKV-bias，attention 加 QK-Norm 稳定大 batch 训练
- MoE 配置：235B 总参 / 22B 激活，128 expert，top-8，无 shared expert；30B-A3B 是同套配方的小版（30B 总参 / 3B 激活）。这条曲线明显在追 DeepSeek V3 的 sparsity ratio（≈ 1/10 激活）
- 预训练 ~36T token，三阶段：通用 → 推理密集 → 长文本（128K window）。MTP 目标作为 auxiliary head
- post-train：thinking / non-thinking 数据混训后再 RL，chat template 里 `<think>` 标签控制是否进入思考模式；这是 Qwen 在 R1 之后的最大动作
- benchmark 上 235B-A22B 在 MATH / AIME / LiveCodeBench 等推理任务追平甚至超过 DeepSeek-V3 的相应配置，体量更小

## 想法

- "一套权重双模式" 是这一年最有商业感的设计：不需要给前端做两个模型路由，token 计费可以按用户拨动 thinking 开关来算，模型也只用维护一套
- 同时铺 dense + MoE 两条产品线说明阿里把推理成本/边缘部署当成战略——0.6B / 1.7B 这种极小 dense 不是 demo，是给端侧/RAG 嵌入做底座的
- 128 expert / top-8 这个比例比 DeepSeek 的 256 / top-8 更保守，sparsity 上没追极致；可能是 expert imbalance 在他们 infra 上还没完全解决

## 一句话

把 V3.1-Terminus 上的 attention 换成 DeepSeek Sparse Attention（DSA）：lightning indexer 选 top-$k=2048$ 个 token 做 attention，复杂度从 $O(L^2)$ 降到 $O(kL)$，128K 上几乎不掉点。

## 方法

V3.2 的架构变化只有 attention 一处。其余 MoE / MLA latent / routing 沿用 V3 体系，从 V3.1-Terminus continue training。

DSA 两步：

- **Lightning Indexer**：用极少的 attention heads（FP8 实现）算每个 query token 对所有前序 token 的 index score $s_{t, j}$。这一段不是真正的 value attention，只是用来排序，所以 head 数和维度都可以砍到很小。
- **Top-$k$ selection**：对每个 query，按 indexer 分数选出最大的 $k=2048$ 个 KV token，再在这 $k$ 个上做完整的 MLA-style attention。论文里训练阶段也是用 $k=2048$。

复杂度：

$$
O(L^2) \;\longrightarrow\; O(L \cdot k + L \cdot L_\text{indexer})
$$

第一项是 selection 后的 attention，第二项是 indexer 本身的 $O(L^2)$ 但常数极小（少 head、FP8）。在 128K context 下推理成本显著下降，benchmark 与 V3.1-Terminus 基本对齐，部分长上下文项还略升。

## 想法

DSA 和 NSA 是同一条路线的两种工程化：NSA 是 train-from-scratch 的三分支门控，DSA 是在已有 dense 模型上"加一个轻量 ranker + 选 top-k"再 continue-train。后者更务实——给定已经花了几千万美金训出来的 V3.1，最低成本拿到长上下文加速。

漂亮的点：indexer 自己就是 attention，但被刻意做得轻（少 head + FP8），等于把"判断重要性"和"实际加权"两件事彻底解耦。这和 NSA 把 importance score 从 compression branch 复用过来是不同思路——DSA 给了 indexer 独立的容量预算，可质疑可调。

没说服我的地方：top-$k=2048$ 是个魔法数，paper 对 $k$ 的扫描其实有限；在更长（>128K）或更分散的检索任务上，这个 $k$ 够不够要看实测。indexer 的训练目标怎么和主任务对齐，paper 写得比较粗。

下一步该看：DSA + NSA 混搭是否可能，以及 indexer 是否可以在 RL post-training 阶段被进一步整形（让它学会按任务调整 top-k 的"软分布"）。
